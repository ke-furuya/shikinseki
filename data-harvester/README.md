# data-harvester — 品質ゲート付き汎用データ収集エンジン

競馬で作ったデータ収集パイプラインを、ドメインから切り離した汎用ツール。**edge-validator の相棒＝「取得→検証」を完結させる上流。**

- `../shikinseki/data_harvester.py` — エンジン本体（標準ライブラリのみ・移植自由）
- `example_mock.py` — 欠陥入り合成データでゲートを実証（移植時はここを書き換える）
- `README.md` — 本ファイル

---

## 普通のスクレイパとの決定的な違い＝完全性ゲート

生データをそのまま使わせない。**"使う前に"、下流（分析）が必要とする項目が揃っているかを
機械検査し、欠損があれば赤で止める。** 「ゴミを入れて、気づかずに分析して、間違った結論を出す」を
物理的に防ぐ。競馬で「採点に必要なA〜Q全項目が揃っているか」を採点前に強制したのと同じ仕組み。

ゲートが捕まえる4つ：
| 検出 | 意味 |
|---|---|
| **必須欠損** | 下流が要る項目がnull/空 → 🔴止める |
| **未知コード** | 生コードがマップに無い（`"?9"`で可視化）→ マップ更新の合図 |
| **取得失敗** | API 502 等で取れなかった → 🔴止める |
| **未収集/余分** | 期待リストと突合（取れているべきが欠けてる/余計なものが混じる） |

「欠損=壊れてる」と「正当に空（古データ・初回・対象外）」を区別するのが肝。
全部一律エラーにすると使い物にならない。

## データの流れ

```
entities（id/パラメータの列）
   │  harvest(entities, fetch_fn)        ← fetch は差し替え自由（HTTP/ブラウザ貼付/ローカル）
   ▼  部分失敗に頑健（1件死んでも全体は止まらない・リトライ）
生データ {id: raw}
   │  map_record(raw, field_spec, code_maps)   ← 生コード→きれいなスキーマ（未知は "?"）
   ▼
きれいなレコード {id: {odds, gate, grade...}}
   │  completeness_gate(records, schema, expected_ids)   ← 下流要件で充足検査＋突合
   ▼
✅PASS → edge-validator のデータ契約に載せて検証へ
⚠️FAIL → 欠損を埋めてから（下流に進ませない）
```

---

## スキーマ定義（＝ゲートの基準＝下流が必要とするもの）

```python
SCHEMA = {
  "required":       ["odds", "gate"],          # 欠ければ🔴止める
  "optional_empty": ["pad", "prev_comment"],   # 空でも可（正当な欠落）
  "code_maps":      {"grade": {"1":"G1","2":"G2",...}},  # 生→値・未マップは "?"
}
```

---

## 別ドメインへの移植（エンジンは不変・3つを書くだけ）

1. **`fetch_fn(entity) -> 生データ`** … APIならURL組み立て＋`http_json()`。1エンティティで複数エンドポイント
   （競馬の card/paddock/training 型）を取って統合する型も書ける
2. **`FIELD_SPEC`／`CODE_MAPS`** … 生のネストパス→出力フィールド、生コード→値
3. **`SCHEMA`** … 下流が必要とするものを required/optional_empty で宣言

`example_mock.py` の末尾に、実APIアダプタの雛形（複数エンドポイントを統合する型）を載せてある。

### 応用例
- どんなサイト/APIからでも、**品質保証付きで**データを取る（取得は万能ニーズ・売れるスキル）
- ログ/メトリクス収集（インフラ）＝必須項目ゲートで「欠測に気づかず集計」を防ぐ
- 外部API連携の取り込み層（ETLの"完全性"を機械保証）

## 実証ログ

`python3 example_mock.py`：わざと欠陥（必須欠損・未知コード・取得失敗・未収集）を仕込んだ合成データで、
ゲートが4種すべてを捕捉して FAIL を返し、修正後に PASS することを確認。

## edge-validator との組み合わせ（取得→検証の完結ユニット）

```
data-harvester（取得・品質保証）  →  edge-validator（効いてる/まぐれ/測れない判定）
```

「どんなソースからでも、品質保証付きでデータを取って、それが本物のシグナルか統計判定するところまで」
——この2本で、信頼できる定量パイプラインの最小完成形になる。これがポートフォリオ／売り物の核。
