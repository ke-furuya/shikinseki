# action-gate — 検証済みの方針を「人為ミスなく実行」するエンジン

競馬プロジェクトで作った「買い目を決定論生成し（手組み禁止）、機械ゲートで
買い忘れ・重複・予算超過を弾く」仕組みを、競馬から切り離した汎用ツール。**4層スタックの最後＝運用層。**

- `../shikinseki/action_gate.py` — エンジン本体（標準ライブラリのみ）
- `example_ops.py` — 決定論生成＋ゲートのブロックを実証
- `README.md` — 本ファイル

---

## 中心思想：人間が手で組むのをやめる

検証で「効く」と分かった方針も、**実行を手作業でやると事故る**（買い忘れ・重複・予算超過・矛盾）。
実際に「本命の穴馬の買い忘れ」で取り逃した経験——その教訓を構造化したもの。

```
方針(policy) + 現在の状態(state)  →  アクションを決定論コード生成（同じ入力→必ず同じ出力）
                                  →  実行前に機械ゲート(invariants)で不変条件を検査
                                  →  違反があれば実行をブロック
```

ミスを「注意力」でなく「構造」で防ぐ。同じstateなら毎回同一のアクションが出る（ルール順にも依存しない）。

---

## 3つの責務

| 関数 | 役割 |
|---|---|
| `generate(policy, state)` | ルール（条件→アクション）から決定論生成。順序非依存・重複排除・監査証跡付き |
| `check(actions, state, invariants)` | 不変条件を適用。`error`は実行ブロック・`warn`は警告 |
| `run(policy, state, invariants)` | generate→check→**通過時のみcommit**（実行可アクションを返す） |

### すぐ使える不変条件ビルダー
`inv_no_duplicate()` / `inv_budget(cost_key, limit)` / `inv_required_fields([...])` /
`inv_not_contradict(key, [(x,y)...])` / `inv_expected_count(expected_fn)`（買い忘れ/出し過ぎ検出）

---

## 別ドメインへの移植（policy と invariants を書くだけ）

`policy = [(ルール名, 条件関数, アクション関数), ...]`、`invariants = [不変条件, ...]` を宣言するだけ。
エンジンは不変。

### 応用例
- **インフラ運用（runbook-as-code・本業直撃）**：state=メトリクス/アラート状況、policy=是正ルール
  （CPU高→スケールアウト等）、invariants=`inv_not_contradict("target",[("scale_up","scale_down")])`/上限ゲート。
  **是正アクションを決定論生成し、矛盾・上限超過を実行前にブロック。** 手作業のオペミスを構造で消す
- **バッチ/デプロイのプリフライト**：実行予定の操作集合を生成→不変条件（必須手順の欠落・順序・上限）で検査
- **通知/施策の自動生成**：条件から送る内容を決定論生成し、重複送信・予算超過をゲート

## 実証ログ

`python3 example_ops.py`：①正常commit ②ルール順シャッフルでも出力不変（決定論の証明）
③予算超過をブロック ④pool 1つ手で落とした不正ポリシーを「件数不一致」で検出ブロック。

## 4層スタック完成

```
data-harvester  →  leak-guard  →  edge-validator  →  action-gate
   ①取得 ✅          ②防御 ✅        ③検証 ✅          ④運用 ✅
  揃ってるか        汚染ないか      本物か判定        ミスなく実行
```

**取得→防御→検証→運用の4層が揃った＝「信頼できる定量システム」の最小完成形。**
どの層も競馬を知らない汎用エンジン＋アダプタ構造。各々が独立に転用でき、繋げば一気通貫のパイプラインになる。
