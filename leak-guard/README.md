# leak-guard — 先読み（リーク）汚染の検出エンジン

競馬プロジェクトのリーク検出（古いレースの馬が後で昇格再登場→勝者逆算可をH/M/Lでフラグ）を
競馬から切り離した汎用ツール。**data-harvester（取得）と edge-validator（検証）の間＝防御層。**

- `../shikinseki/leak_guard.py` — エンジン本体（標準ライブラリのみ）
- `example_leak.py` — わざとリークを仕込んだ合成データで4種の検出を実証
- `README.md` — 本ファイル

---

## なぜ必要か＝リークは「完全なデータ」でも検証を嘘にする

リーク＝予測/採点する時点Tのラベルやsignalが、**T以降にしか得られない情報で汚染**されること。
data-harvester が「データが揃っているか」を保証しても、その揃ったデータが未来情報で汚染されていたら、
edge-validator の判定は嘘になる。**リークのあるエッジは in-sample で光り、実戦（フォワード）で必ず剥がれる。**

これは競馬プロジェクトで実際に何度も起きた：古いレースを採点する時、その馬が後のレースで昇格していると、
「昇格した=過去に勝った/通用した」が逆算でき、無意識に未来の知識が混入する。このリーク検出は
それを H/M/L で機械フラグし、L だけ安全に採点＝人手レビューを H/M に絞った。

---

## 4つの検査（データ契約 group/time/entity/baseline/outcome/features を流用）

| 検査 | 何を捕まえる |
|---|---|
| **① 再登場リーク** | 同じentityが後の時点で"昇格再登場"→過去の結果が逆算可。H/M/L で分級 |
| **② 境界またぎ** | 同じgroupが探索/検証の境界をまたぐ→ホールドアウト検証が壊れる |
| **③ ターゲットリーク** | outcomeを出来すぎなほど当てる特徴量＝答えそのものが混入 |
| **④ 重複** | 同じ(entity,time)が二重 |

### 再登場リークの H/M/L（競馬のリーク検出を一般化）
- **H** = そのgroupの最有力（baseline最小=本命）が後で昇格再登場（結果が逆算可・最危険）
- **M** = 上位top_kが昇格 or 2件以上が昇格
- **L?** = 2件以上が単に再登場（履歴で結果露出の可能性→一瞥）
- **L** = 安全に使える

`level_key`（クラス/ランク等の単調指標）を渡すと「昇格」を厳密判定でき精度が上がる。

---

## 別ドメインへの移植

データ契約に `entity`（主体）と `time` を載せ、可能なら `level`（昇格判定用の単調指標）を足すだけ。
`run_all(records, level_key=..., split_time=..., feature_keys=[...])` で4検査を一括実行。

### 応用例
- **モデル/MLの学習データ監査**：未来の集計値が特徴量に混入（target leakage）していないか。再学習前の必須チェック
- **バックテスト全般**：同一銘柄/ユーザーが学習・検証期間に跨る、未来情報が特徴量に入る、を検出
- **A/Bテスト**：同一ユーザーが両群に出る（汚染）、結果変数が説明変数に紛れ込む

---

## 実証ログ

`python3 example_leak.py`：再登場リーク(H)・境界またぎ・ターゲットリーク・重複を仕込んだ合成データで、
4種すべてを捕捉し、無情報な特徴量は誤検出しないことを確認。

---

## パイプライン上の位置

```
data-harvester（取得・品質保証） → leak-guard（汚染防御） → edge-validator（効いてる/まぐれ/測れない判定）
   ①取得 ✅                          ②防御 ✅                   ③検証 ✅
```

「揃っていて・汚染されておらず・本物か統計判定できる」——この3層で、信頼できる定量評価の核が完成する。
残りは ④運用（決定論生成＋機械ゲートで、検証済みの手順を人為ミスなく回す）。
