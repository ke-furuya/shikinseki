# edge-validator — 「そのエッジは本物か、ただの分散か」汎用判定エンジン

競馬予想の検証プロジェクトで作り込んだ検証手法を、競馬から切り離した汎用ツール。
**「予想（signal）／結果（outcome）／超えるべき相手（baseline）」さえあれば、どんなドメインでも
"効いてる/まぐれ/測れない" を統計的に判定する。**

- `../shikinseki/edge_validator.py` — エンジン本体（標準ライブラリのみ・移植自由）
- `example_sequential.py` — 覗き見の罠と confidence sequence の実証（偽陽性49%→0%）
- `README.md` — 本ファイル（データ契約＋移植手順＋方法論）

---

## なぜこれが価値あるか

世の中、予測・施策・モデルを作る人は山ほどいる。でも **「その数字、本当に効いてる？ それとも
良く見えてるだけ？」を統計的に答えられる人はほとんどいない。** 多くが「精度95%！」「回収率200%！」を
鵜呑みにして、**分散の揺れ（まぐれ）を実力と勘違いする。**

このエンジンは、その勘違いを構造的に防ぐ。元になった競馬プロジェクトでは、
**「回収率156%の発見」が、母数を増やしたら98%に回帰する蜃気楼だった**ことを、このロジックで暴いた。

## データ契約（1行＝1つの判断/観測）

| フィールド | 型 | 意味 |
|---|---|---|
| `group` | str | この行が属するイベント（同一groupは相関＝再標本化の単位）。例: レースID/実験ID/日付/セッションID |
| `time` | str | 時系列ホールドアウト分割用。ソート可能なら何でも（ISO日付推奨） |
| `baseline` | float | **超えるべき相手**の予測。市場確率/対照群の率/現行システム/コンセンサス。0〜1の確率か単調スコア |
| `outcome` | float | 実際に起きたこと（1/0の的中、または数値リターン） |
| `cost` | float | アクションのコスト/賭け金（ROI用・既定1） |
| `payoff` | float | アクションした時の粗リターン（既定=outcome）。ROI=payoff/cost |
| `entity` | str | リーク検査用の主体（馬/銘柄/ユーザー/患者…） |
| （任意） | — | 残差スキャン用の特徴量を任意キーで（`maru`, `pad` 等） |

## エンジンが答える問い

| 関数 | 問い | 防ぐ間違い |
|---|---|---|
| `roi_ci()` | 価値（ROI）と、まぐれを除いた**信頼区間**は？ | 点推定だけ見て分散に騙される |
| `mean_ci()` | 任意の量（正解率・**A/B差**・効果量）の平均と信頼区間は？ | A/B比較で点推定の勝者を鵜呑み |
| `power_required()` / `power_required_mean()` | **事前指定した最小効果(MDE)** を証明するのに何件要るか＝測れるのか | 永遠に測れない差を追う／観測差を使うpost-hoc powerで過小評価 |
| `confidence_sequence()` / `sequential_scan()` | **増やしながら何度も覗いても妥当**な信頼区間（anytime-valid・厳密保証はσ明示時） | 「有意になるまで覗いて止める」optional stoppingの自己欺瞞 |
| `residual_scan()` | 特徴量は**baselineを超えて**結果を予測するか | 「相手が既に知ってる情報」を独自エッジと誤認 |
| `holdout_scan()` / `staged_scan()` | 探索/検証（/確認）に分け**再現したものだけ**残す | in-sampleマイニングの蜃気楼／選択後の勝者の呪い |
| `leak_check()` | signalが**未来情報で汚染**されてないか | リークで過大評価（実戦で消える） |
| `pbo_cscv()` / `deflated_sharpe_ratio()` / `probabilistic_sharpe_ratio()` | **過学習確率**・試行数で割引いた本物度（クオンツ標準） | N個試して最良を拾う選択バイアス（156%の蜃気楼） |
| `purged_kfold_indices()` | 時系列の**purged＋embargo**分割 | 隣接サンプルからの情報漏れ |
| `cohens_kappa()` | **アノテータ間一致**（ラベル品質の前提） | 不安定な「正解」で精度を語る |
| `decision_value_ab()` / `value_of_information()` | 「本物か」→**行動すべきか／集める価値があるか** | 有意性だけ見てコスト・意思決定を無視 |

> **estimand を宣言する：** `staged_scan(..., estimand="...")` で「何を推定しているか（予測残差か因果効果か）」を一文で固定。混同が一番の事故。

> **固定N vs 逐次**：Nを先に決めて1回だけ判定するなら `roi_ci`/`mean_ci`（90%CI）でよい。
> 「灰色ならデータを足して再確認」を繰り返すなら `confidence_sequence`（覗き見OKだが固定Nより広い）。
> 混同して「固定CIで何度も覗いて有意で止める」と、真の効果ゼロでも高確率で偽陽性（`example_sequential.py`で49%を実証）。

## 別ドメインへの移植手順（エンジンは不変・アダプタだけ書く）

1. **生データ → データ契約** に変換する `load_*()` を書く（fetch関数とスキーマだけ書けばよい）。
   - `baseline` に「超えるべき相手」を入れる（A/Bなら対照群率、トレードなら市場、モデルなら現行版）
   - `outcome` に結果、`cost`/`payoff` に価値（あれば）
2. 検証したい特徴量を `(ラベル, 述語関数)` のリストで定義
3. `holdout_scan()` で **再現したものだけ** を拾う。`roi_ci()`/`power_required()` で価値と測定可能性
4. survivors が出たら、それが「相手が取りこぼしてる本物の候補」。次は**前向き（フォワード）検証**で確定

### このエンジンが効くドメインの条件
- **baseline（超えるべき相手）が定義できる** … これが肝。相手のいない「ただの予測」には使えない
- **相手が"非効率"なほど妙味が出る** … 競馬が負けたのは市場が効率的すぎたから。
  逆に相手がコンセンサス/ヒューリスティック/現行システムで**穴がある**領域ほど、survivorsが出やすい

### 応用例
- **A/Bテスト/施策効果**: baseline=対照群、outcome=コンバージョン。「この施策、本当に効いた？分散？」
- **モデル/プロンプト評価**: baseline=現行モデル、outcome=正解。「新版は本当に良い？評価セットのまぐれ？」
- **業務の予兆検知（インフラ/SRE）**: baseline=現行アラート、outcome=実障害。「このルールは誤検知を減らすか」
- **トレード/予測市場**: baseline=市場価格。※競馬と同じく効率的市場は厳しい＝相手の非効率を見極めてから

## 方法論の原則（競馬プロジェクトの教訓を一般化したもの）

1. **点推定でなく信頼区間で判断する。** 「ROI 156%」は的中数本に依存して乱高下する。CI下限がプラス圏かを見る
2. **大きい数字ほど疑う。** 最良に見える結果は、たいてい分散が一番大きい（＝一番測れない）構造
3. **探索と検証を必ず分ける。** in-sampleなら何かは必ず光る。out-of-sample再現だけが本物
4. **試行回数を数える（多重比較）。** 20個試せば偶然1個は"有意"。試行数を申告しないエッジは信じない
5. **測れるかを先に問う（検出力）。** 真の差が小さいと、現実離れした件数が要る＝証明不能。追う前に判定
6. **baselineを超えるかで測る。** 「当たった」ではなく「相手より当たった」。相手が既に知ってる情報は価値ゼロ
7. **リークを疑う。** signalに未来情報が一滴でも混じると過大評価。実戦（フォワード）で必ず剥がれる
8. **止めどきを結果で決めない（覗き見の罠）。** 「有意になった瞬間に止める」と、真の効果ゼロでも偶然いつか線を越える。Nを先に決めて1回判定するか、新データで答え合わせするか、anytime-validなCSを使う

> この8原則は、いずれも「自分を騙さない」ための——ありふれているが、守り続けるのが一番難しい規律だ。
> 「正直に効果を見極める」こと自体が希少なスキルであり、これがこのツールの中核IP。

## 実証ログ（リファレンス・ケーススタディ）

`python3 example_advanced.py` の出力＝汎用エンジンが自己検証つきで全機能を実行：
- **市場効率テスト**: 公開特徴量10角度を探索/検証分割 → out-of-sample再現は **0個**（市場が全て織込済み）
- **単勝◎**: ROI 80% / CI 72〜89% = **頑健にマイナス**（＝控除率20%の床ぴったり）

この「勝てないことを検出力付きで証明した記録」自体が、ツールの信頼性の証明になっている。
